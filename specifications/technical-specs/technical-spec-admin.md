# Technical Spec — Админ-панель и услуги в БД

Техническая реализация админ-панели управления услугами (продуктовое описание — `../feature-specs/feature-spec-admin.md`). Покрывает: перенос услуг из файлов в БД, схему таблицы, чтение на сайте, авторизацию, RLS, авто-перевод, генерацию slug, ревалидацию, env и затронутые места.

Связанные документы: `technical-spec.md` (сквозная техника), `technical-spec-booking.md`, `../feature-specs/feature-spec-admin.md`, `../feature-specs/feature-spec-services.md`, `../feature-specs/feature-spec-chat.md`.

---

## 1. Суть изменения

Сегодня услуги живут в `content/services.ts` (статический TypeScript), страницы услуг — SSG. Чтобы владелец управлял услугами сам, услуги переезжают в **таблицу Supabase `services`**, а весь сайт (и чат) читает их оттуда. Остальной контент (`site`, `doctors`, `reviews`, `faq`) в v1 **остаётся в файлах**.

Это вторая таблица в проекте после `bookings` и первое использование **Supabase Auth** (для входа в админку). Меняется принцип «весь контент в файлах» из `technical-spec.md` §4 — только для услуг.

## 2. Схема таблицы `services`

Скалярные поля — колонками, вложенное и многоязычное — в `JSONB`. Многоязычное значение хранится как `{ "ru": …, "en": …, "tr": … }`.

| Колонка | Тип | Назначение |
|---|---|---|
| `id` | `uuid` PK, `default gen_random_uuid()` | Внутренний идентификатор (для маршрутов админки) |
| `slug` | `text` unique not null | Часть URL `/services/<slug>`, генерируется, неизменна |
| `price_from` | `integer` not null | Цена «от», USD |
| `price_to` | `integer` not null | Цена «до», USD |
| `show_on_home` | `boolean` not null default `false` | Показывать ли в витрине на главной |
| `sort_order` | `integer` not null default `0` | Порядок вывода на сайте |
| `name` | `jsonb` not null | Название (локализованное) |
| `short_description` | `jsonb` not null | Короткое описание (локализованное) |
| `full_description` | `jsonb` not null | Полное описание (локализованное) |
| `trip` | `jsonb` not null | `{ visits: {value,caption}, days: {value,caption} }` (локализованное) |
| `stages` | `jsonb` not null default `'[]'` | Массив `{ title, description }` (локализованное) |
| `sub_procedures` | `jsonb` not null default `'[]'` | Массив `{ name(лок.), price_from, price_to }` |
| `faq` | `jsonb` not null default `'[]'` | Массив `{ q, a }` (локализованное) |
| `related_doctors` | `text[]` not null default `'{}'` | Идентификаторы врачей (врачи в v1 — в файлах) |
| `created_at` | `timestamptz` default `now()` | — |
| `updated_at` | `timestamptz` default `now()` | Обновляется триггером при изменении |

Ограничения уровня БД: `price_to >= price_from`; такие же check'и на цены под-процедур — на уровне валидации перед записью (внутри JSONB БД не проверяет).

**Миграции и сид.** Таблица создаётся миграцией (`supabase/migrations/`, см. CLAUDE.md). Отдельной seed-миграцией в неё переносятся текущие 8 услуг из `content/services.ts` — один к одному, чтобы ничего не потерять. После миграции — регенерация типов (`supabase gen types … > lib/db/types.generated.ts`).

## 3. Чтение услуг на сайте

Появляется слой данных `lib/services/` (напр. `getServices()`, `getServiceBySlug()`, `getHomeServices()`), который читает из Supabase **на сервере под service-role** — тем же ключом, что и `bookings` (anon-ключ для чтения данных по-прежнему не нужен). Все нынешние импорты из `content/services.ts` заменяются вызовами этого слоя:

- `app/[locale]/services/page.tsx`, `app/[locale]/services/[slug]/page.tsx`;
- `components/sections/ServicesGrid.tsx`, `ServiceCard.tsx`, `ServicesPreview.tsx` (последний — `show_on_home`, до 5 по `sort_order`; сетка витрины делается адаптивной под 1..5, текущий `lg:grid-cols-5` перестаёт предполагать ровно 5 — деталь раскладки идёт через design-map);
- `lib/utils/sitemap.ts`;
- `lib/chat/context.ts` (контекст чата), `lib/chat/parseBookingMarker.ts`;
- **форма записи**: `components/booking/BookingForm.tsx` (список услуг) и `lib/validation/booking.ts` (см. §7).

`content/services.ts` после переноса удаляется; тип `Service` (`types/service.ts`) сохраняется как форма данных, но `slug` становится свободной строкой (текущий union `ServiceSlug` убирается — он больше не фиксирован).

## 4. SSG, динамические страницы и ревалидация

- `generateStaticParams` для `/services/[slug]` читает slug'и **из БД** на этапе сборки.
- `dynamicParams = true` (по умолчанию) — услуга, добавленная после сборки, рендерится по запросу, без передеплоя.
- Чтение услуг помечается тегом кэша (`unstable_cache` / `revalidateTag('services')`). После любой записи в админке Server Action вызывает `revalidateTag('services')` (и при необходимости `revalidatePath`) — публичные страницы и sitemap обновляются. Допустима небольшая задержка (см. feature-spec §6).

## 5. Авторизация (Supabase Auth)

- Вход — **Supabase Auth** (email + пароль). Единственный админ-пользователь создаётся вручную (Supabase dashboard / CLI), публичной регистрации нет.
- Сессия — cookie-based через **`@supabase/ssr`** (стандартный помощник Supabase для App Router; новая зависимость, обоснование — официальный путь для серверной авторизации в Next). Браузерному клиенту для логина нужен **publishable (anon) ключ** — это осознанное отклонение от «anon-ключ не используется»: он применяется **только для потока авторизации админки**, не для чтения контента.
- Защита маршрутов `/admin/*`: проверка сессии в `proxy.ts` (middleware) и/или в layout админки; нет сессии → редирект на `/admin/login`.
- `/admin` живёт **вне языковых префиксов** — middleware исключает `/admin` из локального роутинга и из редиректа `/` → `/ru`.
- Ограничение попыток входа — на уровне Supabase Auth (встроенный rate-limit); captcha нет.

## 6. Запись из админки

- Все мутации (создать/изменить/удалить услугу) — через **Server Actions**. Server Action — публичный эндпоинт: каждый action **самостоятельно** проверяет активную сессию админа (не полагаясь на то, что страница защищена), и только потом пишет в `services` **под service-role** — единый с проектом принцип «мутации под service-role» (`technical-spec.md` §9).
- Валидация — Zod-схема услуги в `lib/validation/` (обязательные RU-поля, цены ≥ 0 и «от» ≤ «до», корректность вложенных массивов). Схема общая для формы и сервера.
- Удаление — с проверкой и подтверждением на клиенте; необратимо (soft-delete/история в v1 нет).

## 7. RLS

- На `services` **включён RLS, разрешающих политик нет** — как и у `bookings`. Чтение сайтом и запись из админки идут под service-role, который RLS обходит на сервере. Anon/authenticated-доступа к таблице нет.
- Таблицы Supabase Auth (`auth.*`) управляются самим Supabase.

## 8. Авто-перевод

- При создании/редактировании владелец заполняет **русские** поля. Server Action перед записью вызывает OpenAI (клиент `openai` уже в проекте) и переводит локализованные поля RU → EN и RU → TR одним структурированным запросом (ответ — JSON по полям).
- Результат пишется в `en`/`tr` ключи JSONB и **доступен для ручной правки** в форме.
- Сбой перевода не блокирует сохранение: услуга сохраняется с заполненным RU (и пустыми/прежними EN·TR), перевод можно перезапустить. На сайте отсутствующий перевод → фолбэк на RU.
- **Сид не переводится заново:** перенесённые из файла 8 услуг сохраняют свои готовые EN/TR; авто-перевод применяется только к новым или изменённым русским текстам.
- **Самый хрупкий участок** — перевод вложенных массивов (этапы, под-процедуры, FAQ): большой структурированный JSON, нужно сохранить и структуру, и грамматику подписей логистики. Реализуется через structured output; при сбое — фолбэк выше.

## 9. Генерация slug

- Slug формируется из **английского названия** (после авто-перевода): нижний регистр, латиница, пробелы → дефисы, мусорные символы убираются (`«Dental Implants»` → `dental-implants`).
- Уникальность гарантируется: при коллизии добавляется числовой суффикс (`-2`, `-3`).
- Задаётся один раз при создании, далее **read-only** (feature-spec §4).

## 10. Затронутые места (чек-лист реализации)

- Booking: `BOOKING_SERVICE_VALUES` (зашитый enum в `lib/validation/booking.ts`) убирается; список услуг для формы и проверка `service` берутся из БД. Допустимое множество — текущие slug'и + `other`; проверка значения переносится на сервер (где доступен актуальный список). Тест `booking.test.ts` обновляется.
- `types/service.ts`: `slug: string` вместо union; добавляется поле `showOnHome` (и `sortOrder`, `id`) в форму данных.
- Чат: `lib/chat/context.ts` собирает услуги из слоя данных (async), а не из статического импорта.
- Заявки `bookings` хранят выбранную услугу как slug-текст (без внешнего ключа) — исторический след; удаление услуги старые заявки не трогает.
- Услуги редактируются только через `/admin` (не через dashboard Supabase) — прямая правка JSONB обходит валидацию и авто-перевод и может испортить структуру.

## 11. Environment Variables (добавления к `technical-spec.md` §9)

| Переменная | Назначение |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | URL Supabase для браузерного auth-клиента |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Publishable (anon) ключ — **только** для потока авторизации админки |

`SUPABASE_SERVICE_ROLE_KEY` и `OPENAI_API_KEY` уже есть (service-role — чтение/запись услуг; OpenAI — переводы). Обновляется `.env.example`.

## 12. Зависимости

- `@supabase/ssr` — cookie-сессии Supabase Auth в App Router. Единственная новая зависимость; обоснование — официальный путь серверной авторизации Supabase для Next.js.

## 13. Тестирование

- Zod-схема услуги — позитивный/негативный кейсы (обязательные поля, цены, «от» ≤ «до»).
- Генерация slug — slugify + разрешение коллизий (чистая функция).
- Обновление `booking.test.ts` под динамический список услуг.
- Авто-перевод и Server Actions с сетью — не покрываем (в духе `technical-spec.md` §11: только извлекаемая чистая логика).

## 14. Этапность реализации (отдельные планы/PR)

1. **Хребет:** миграция `services` + seed из файла + слой `lib/services/` + перевод всех потребителей (включая booking) на чтение из БД + ревалидация. Снаружи без изменений.
2. **Доступ:** `@supabase/ssr`, Supabase Auth, защита `/admin/*`, экран входа, пустой каркас панели.
3. **CRUD:** формы создания/редактирования/удаления, Zod-валидация, авто-перевод, генерация slug, `revalidateTag`.
