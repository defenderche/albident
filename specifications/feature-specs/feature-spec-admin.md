# Feature Spec — Админ-панель (`/admin`)

Закрытая часть сайта для **владельца клиники**: он входит по логину и паролю и сам управляет содержимым услуг — добавляет новые, редактирует и удаляет существующие, без участия разработчика. Это превращает услуги из «зашитого в код» контента в управляемый владельцем.

В учебном/портфолио-контексте (`../global-spec.md` §1) цель админки — продемонстрировать полноценный full-stack: авторизация, защищённые маршруты, CRUD и работа с базой данных.

Техническая реализация (таблица, авторизация, RLS, ревалидация, env) — в `../technical-specs/technical-spec-admin.md`.

---

## 1. Кто пользуется

- **Владелец клиники** (в учебном MVP — владелец проекта, см. `../global-spec.md` §16). Единственная роль с доступом. Ролей/прав внутри панели нет — вошедший пользователь может всё в рамках управления услугами.
- Пациенты и обычные посетители сайта админку не видят и про неё не знают.

## 2. Доступ и вход

- Панель живёт по адресу `/admin` (вне локалей — интерфейс админки только на русском, языковых версий у самой панели нет; переключатель локалей сайта на неё не распространяется).
- Открытие `/admin` без активной сессии → **экран входа** (`/admin/login`): поля «email» и «пароль», кнопка «Войти».
- Неверные данные → сообщение об ошибке, без деталей («неправильный email или пароль»).
- Попытки входа ограничиваются (защита от перебора пароля); captcha при этом не используется — как и в формах MVP.
- Успешный вход → попадание в панель; сессия удерживает пользователя залогиненным между визитами до выхода или истечения срока.
- Любой защищённый маршрут админки при отсутствии сессии перенаправляет на `/admin/login`.
- В панели есть кнопка **«Выйти»**, завершающая сессию.
- Регистрации нет. Учётная запись владельца создаётся вручную (см. техспеку). Восстановления пароля в v1 нет.

## 3. Структура панели

| Маршрут | Назначение |
|---|---|
| `/admin/login` | Экран входа |
| `/admin` | Список услуг + кнопка «Добавить услугу» |
| `/admin/services/new` | Форма создания новой услуги |
| `/admin/services/[id]/edit` | Форма редактирования существующей услуги |

**Список услуг (`/admin`)** показывает все услуги (название RU, slug, диапазон цены), у каждой — действия «Редактировать» и «Удалить». Порядок отражает порядок вывода на сайте.

Панель — **desktop-first** (владелец работает с большой формой за компьютером), но остаётся рабочей на мобиле.

## 4. Что владелец редактирует в услуге

Владелец заполняет услугу **на русском**. Английскую и турецкую версии система **переводит автоматически** при сохранении (через OpenAI — тот же сервис, что у чата) и кладёт в соответствующие поля. Переводы остаются **редактируемыми**: машинную версию можно поправить вручную через языковые вкладки (соответствует `../global-spec.md` §15 — машинный перевод с ручной вычиткой). Так у владельца минимум работы — он пишет на одном языке, — а на сайте не остаётся пустых языков.

Поля услуги:

- **Название** — русский; EN/TR — авто-перевод.
- **Короткое описание** — для карточки услуги.
- **Полное описание** — для страницы услуги.
- **Цена «от» и «до»** — в USD, числом (без локализации, едина для всех языков; см. `../global-spec.md` §9). Правило: «от» ≤ «до».
- **Логистика поездки** — число визитов и дней в Стамбуле + подпись к каждому (см. `feature-spec-service-page.md`).
- **Этапы лечения** — упорядоченный список (заголовок + описание). Можно добавлять, удалять, переупорядочивать. Допустимо оставить пустым — тогда блок этапов на странице услуги не показывается.
- **Под-процедуры** — список: название + цена «от/до» (USD). Необязательный.
- **FAQ услуги** — список вопрос/ответ. Необязательный.
- **Связанные врачи** — выбор из существующих врачей (врачи в v1 остаются в файлах; владелец их не редактирует, только отмечает связь — выбрать можно лишь реально существующего врача).
- **Показывать на главной** — флаг: попадает ли услуга в витрину на главной странице. Главная показывает до **5** отмеченных услуг по порядку (`sort_order`); сетка витрины подстраивается под любое их число (1..5), не ломаясь при отличии от пяти.

**Идентификатор (slug)** владелец не вводит. Система генерирует его автоматически из английского названия (напр. «Dental Implants» → `dental-implants`); он становится частью адреса `/services/<slug>` и **после создания не меняется** — чтобы не ломать сохранённые ссылки и адреса страниц.

Перед сохранением данные проверяются: русские обязательные поля заполнены, цены — неотрицательные числа и «от» ≤ «до», сгенерированный slug уникален. При ошибке форма показывает, что именно не так, и не сохраняет. Если авто-перевод не удался (сбой сервиса) — услуга всё равно сохраняется на русском, перевод можно запустить повторно или вписать вручную.

## 5. Действия над услугами

- **Создать** — заполнить форму `/admin/services/new`, сохранить → услуга появляется в списке и на сайте.
- **Редактировать** — изменить поля на `/admin/services/[id]/edit`, сохранить.
- **Удалить** — из списка, с подтверждением («Удалить услугу „…“? Действие необратимо»). После удаления услуга исчезает с сайта; её страница `/services/<slug>` начинает отдавать 404. Уже оформленные заявки на эту услугу не затрагиваются (в заявке услуга записана текстом, как исторический след).

Сохранение создания/редактирования занимает пару секунд — идёт авто-перевод на EN/TR; форма показывает индикатор и не даёт отправить дважды.

## 6. Как изменения отражаются на сайте

Услуги хранятся в одном месте (БД), откуда их читают **и публичный сайт, и чат-ассистент**. Поэтому после сохранения в админке:

- обновляются карточки на `/services` и главной (по флагу «на главной»), страница услуги `/services/<slug>`, `sitemap.xml`;
- **форма записи `/booking`** — выпадающий список услуг и проверка допустимых значений берутся из того же источника (раньше список был зашит в коде — теперь читается из БД), поэтому новая услуга сразу доступна для выбора при записи;
- **чат-ассистент** при следующих ответах знает про изменения (собирает контекст из того же источника, см. `feature-spec-chat.md`);
- ничего дублировать вручную не нужно.

Услуги меняются **только через `/admin`**. Править их напрямую в dashboard Supabase не следует — это обходит авто-перевод и проверки и может сломать структуру данных. (В отличие от заявок `bookings`, которые менеджер как раз читает в dashboard.)

Обновление публичной части после сохранения — через ревалидацию (детали — в техспеке). Допустима небольшая задержка перед тем, как изменение станет видно посетителю.

## 7. Что НЕ входит в v1

Сознательно отложено:

- Управление **врачами, отзывами, общим FAQ, контактами** — остаются в файлах (`content/*.ts`). Админка v1 — только услуги.
- Загрузка изображений через панель.
- Несколько пользователей, роли и права, аудит действий.
- Восстановление/смена пароля из интерфейса, двухфакторная аутентификация.
- Черновики/предпросмотр, история версий, откат изменений.
- Локализация интерфейса самой панели (только русский).

Эти пункты могут войти в следующие версии, но в v1 их нет — чтобы получить один законченный, надёжный кейс (услуги) целиком.

## 8. Индексация

Админка закрыта от поисковых систем: маршруты `/admin/*` не индексируются (`noindex`), в `sitemap.xml` не попадают, в `robots.txt` запрещены.
