# Feature Spec — Запись (`/booking`)

Главное целевое действие сайта. Форма заявки на обратный звонок: пациент оставляет данные, менеджер связывается с ним через WhatsApp и подтверждает запись.

Реальной интеграции с расписанием врачей и автоматического бронирования слотов **нет**.

Техническая реализация (валидация, Server Action, схема БД, отправка email) — в `../technical-specs/technical-spec-booking.md`.

---

## 1. Точки входа

- Прямой переход на `/booking`.
- Кнопка «Записаться» в шапке (header).
- Кнопки «Записаться» с других страниц — главной, страниц услуг и т.д.
- **Deeplink с предзаполненной услугой:** `/booking?service=<slug>` — открывает форму с уже выбранной услугой в dropdown. Используется при переходе со страницы конкретной услуги.

## 2. Поля формы

| Поле | Тип | Обязательное | Назначение |
|---|---|---|---|
| Имя | text | ✓ | Как обращаться к пациенту |
| Телефон (WhatsApp) | tel | ✓ | Основной канал связи |
| Город / страна | text | — | Контекст для менеджера, особенно для дентал-туристов |
| Услуга | dropdown | — | Чтобы менеджер сразу знал, о чём речь |
| Предпочтительное время | text | — | Удобные день/время для звонка |
| Комментарий | textarea | — | Дополнительная информация от пациента |
| Согласие на обработку ПДн | checkbox | ✓ | Со ссылкой на `/privacy`. Обязательный, поскольку заявки сохраняются в БД (см. `../global-spec.md §14`) |

**Список услуг в dropdown:** Терапия, Хирургия, Имплантация, Протезирование, Ортодонтия, Эстетика, Гигиена / профилактика, Пародонтология, Другое.

## 3. Валидация

- **Имя:** не пустое, минимум 2 символа. Ошибка: «Введите имя».
- **Телефон:** не пустой, минимум 10 цифр (плюс может содержать `+`, пробелы, дефисы). Ошибка: «Введите корректный номер телефона».
- Ошибки показываются под соответствующим полем.
- Остальные поля — без жёсткой валидации, могут оставаться пустыми.

## 4. Канал доставки заявки

Заявка обрабатывается в два этапа:

1. **Запись в БД (Supabase, таблица `bookings`).** Первичный канал — все заявки сохраняются и доступны менеджеру через дашборд Supabase. Структура таблицы — `../technical-specs/technical-spec-booking.md §2`.
2. **Email-уведомление клиники.** Отправляется только если запись в БД прошла успешно. «Мгновенный сигнал» менеджеру о новой заявке.

**Содержимое письма:**
- Поля формы: имя, телефон, город/страна, услуга, время, комментарий.
- Дата и время подачи заявки.
- Язык сайта, с которого пришла заявка (TR / EN / RU).

Согласие на ПДн в письмо не выносится — чекбокс обязательный, поэтому это поле всегда `true` и не несёт информации для менеджера. Аудит-след согласия хранится в БД (поле `consent` в таблице `bookings`).

**Тема письма:** «Новая заявка с сайта — [Имя]».

**Если БД упала:** заявка не сохраняется, пациент видит сообщение об ошибке (см. §5.4). Email НЕ отправляется.

**Если БД прошла, а email упал:** пациент видит успех. Заявка сохранена в БД, менеджер увидит её в дашборде. Ошибка отправки логируется.

**Timeout.** Server Action на Vercel имеет дефолтный лимит 10 секунд. Если БД не ответила за это время — пациент видит Error (см. §5.4), заявка считается несостоявшейся. Автоматического retry на сервере нет.

**Логирование.** Ошибки записи в БД и отправки email логируются через `console.error` — попадают в Vercel logs. Внешнего мониторинга и алертов нет (см. `../global-spec.md §18`).

## 5. Состояния формы

### 5.1 Idle (исходное)

Форма доступна для заполнения. Кнопка «Отправить» активна.

### 5.2 Loading (отправка)

После нажатия «Отправить»:
- Кнопка становится неактивной.
- Текст кнопки меняется на «Отправляется…».

### 5.3 Success (успешная отправка)

Форма заменяется inline-сообщением:

> «Спасибо, мы свяжемся с вами в WhatsApp в ближайшее время.»

Возврат к форме — обновлением страницы или повторным переходом на `/booking`.

### 5.4 Error (ошибка отправки)

При сбое сети или сервера:
- Над формой появляется сообщение: «Не удалось отправить заявку. Попробуйте ещё раз или свяжитесь с нами через WhatsApp.»
- Заполненные данные **сохраняются** в полях.
- Кнопка снова активна, можно повторить отправку.
- Автоматического retry нет — повторная отправка только по клику пациента.

## 6. Антиспам

Защиты нет (учебный проект). В реальной эксплуатации этот пункт пересматривается.
