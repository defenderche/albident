# Albident

Сайт стоматологического центра «Albident» в Стамбуле (район Şişli). Next.js + TypeScript.

Текущий статус: спецификация завершена, Next.js заскаффолжен с i18n / Vitest / CI. См. `roadmap/roadmap.md` для прогресса по областям.

## Команды

- `npm run dev` — dev-сервер на http://localhost:3000 (корень редиректит на `/ru`)
- `npm run build` — production-сборка
- `npm run start` — запуск production-сборки
- `npm run lint` — ESLint
- `npm run format` — Prettier (форматирование), `npm run format:check` — проверка без правки
- `npm test` — Vitest
- `npx tsc --noEmit` — проверка типов без эмита

## Стек

- Next.js (App Router) + TypeScript (strict mode)
- React 19+
- Tailwind CSS
- ESLint (стандартный конфиг Next.js) + Prettier
- Supabase (Postgres) + `@supabase/supabase-js`
- OpenAI API (`gpt-4o-mini`) — чат-ассистент
- Vercel KV — счётчики лимитов чата
- Vitest — юнит-тесты в критичных местах

## Соглашения по коду

- Отступ — 2 пробела.
- Компоненты — `PascalCase.tsx`. Утилиты и хуки — `camelCase.ts`.
- Серверные компоненты по умолчанию; `"use client"` — только при необходимости интерактива.
- Изображения — через `next/image`. Шрифты — через `next/font`.
- Каждая страница экспортирует `metadata` (`title`, `description`).
- Никаких секретов в коде. Переменные окружения — `.env.local` (gitignored), пример — `.env.example` (в репо).

## Тестирование

В MVP — **только юнит-тесты в критичных местах**. Цель — ловить регрессии там, где ошибка дорого стоит.

Покрываем:

- **Валидация форм** (Zod-схемы в `lib/validation/`) — позитивный и негативный кейс для каждой схемы.
- **Лимиты чата** (счётчик Vercel KV в `lib/chat/`) — логика «можно/нельзя» при разных значениях счётчика.
- **Чистые утилиты** (`lib/utils/`) — где есть нетривиальная логика.

Не покрываем в MVP:

- UI-компоненты и визуальные проверки.
- E2E-сценарии (Playwright / Cypress).
- Server Components — тестируем только извлекаемую логику, не JSX.

Стек: **Vitest**. Тесты лежат рядом с кодом: `<name>.test.ts` рядом с `<name>.ts`. Запуск: `npm test`.

## CI / GitHub Actions

На каждом PR в `main` GitHub автоматически запускает три проверки:

- `npm run lint` — ESLint.
- `npx tsc --noEmit` — TypeScript-проверка типов.
- `npm test` — Vitest.

Если хотя бы одна падает — на PR появляется красный значок, кнопка Merge недоступна.

Конфиг — `.github/workflows/ci.yml` (создаётся после скаффолда Next.js).

Pre-commit hooks (husky / lint-staged) **не используются** — Actions покрывают всё нужное на серверной стороне.

Дополнительно Vercel при создании PR делает preview-build — отдельная проверка, что проект собирается.

## База данных (Supabase)

### Миграции

Любое изменение схемы — только через миграции, **не через dashboard Supabase напрямую**.

- SQL-файлы миграций лежат в `supabase/migrations/`.
- Создание: `supabase migration new <name>` (Supabase CLI).
- Применение: `supabase db push` (локально / на dev / на prod).
- Каждая миграция — атомарное изменение (новая таблица, новый столбец, индекс, RLS-политика).

### RLS (Row Level Security)

**Включён на всех таблицах без исключений.** Это страховка: даже если ключ утечёт, без явных политик никто извне ничего не сможет.

- Доступ к **данным** (`bookings`, `services`) — только серверный, под `SUPABASE_SERVICE_ROLE_KEY`; клиентских обращений к данным нет. Контент сайта живёт в `content/*.ts`, **кроме услуг** — они в БД (см. `specifications/technical-specs/technical-spec-admin.md`).
- Anon-ключ используется **только** для входа в админ-панель `/admin` через Supabase Auth — не для чтения/записи данных.
- Все мутации данных — через Server Action под `SUPABASE_SERVICE_ROLE_KEY`. Service-role обходит RLS на сервере.
- Никаких разрешающих политик для `anon`/`authenticated` на таблицах данных не пишем — RLS включён, политик нет, доступ только под service_role.
- Менеджер клиники читает заявки через Supabase dashboard под отдельным паролем (через UI, не через ключ из проекта). Услугами владелец управляет через `/admin`, не через dashboard.

### TypeScript-типы

Типы таблиц генерируются из реальной схемы:

```bash
supabase gen types typescript --linked > lib/db/types.generated.ts
```

Запускается после каждой миграции, чтобы код знал актуальную структуру. Файл `lib/db/types.generated.ts` — **генерируемый, руками не правится**.

## Соглашения по контенту

- **Язык общения с пользователем — русский.**
- Контент сайта — на трёх языках: TR (турецкий), EN (английский), RU (русский). Язык по умолчанию — RU.
- URL-схема локалей: path-based (`/ru/...`, `/en/...`, `/tr/...`).
- Идентификаторы кода и имена файлов — английский.
- **Сообщения git-коммитов — русский**, без conventional commits префиксов (`feat:`, `fix:` и т.п.). Пример: «Добавлена форма записи».

## Особенности проекта

- **Универсальная стоматология**, средний сегмент. Стартовый набор — 8 услуг: терапия, хирургия, имплантация, протезирование, ортодонтия, эстетика, гигиена, пародонтология (без детской). Далее услуги управляются владельцем через админ-панель `/admin` (хранятся в БД).
- **Аудитория** — местные пациенты + дентал-туристы из России/СНГ и Европы.
- **Главное действие** — онлайн-запись через форму на `/booking`. Заявка сохраняется в Supabase + дублируется email'ом клинике. Менеджер связывается с пациентом через **WhatsApp**, заявки просматривает в дашборде Supabase. См. `specifications/technical-specs/technical-spec-booking.md`.
- **Чат-ассистент** на всех страницах (плавающая кнопка в углу), отвечает на вопросы о клинике в рамках жёстких медицинских границ. См. `specifications/feature-specs/feature-spec-chat.md`.
- **Фото реальных людей не используются.** Врачи на странице «Команда» отображаются как рисованные аватары (стилизованные AI-иллюстрации, не фото реальных людей) + ФИО + роль + достижения. Fallback на силуэт, если аватар отсутствует. См. `global-spec.md §12` (пересмотрено 2026-06-09).
- **Соцсети:** Instagram, WhatsApp, Telegram (без Facebook).
- **Legal-заглушки** для будущего наполнения: страница `/privacy` с placeholder-текстом, ссылка в подвале, **обязательный** чекбокс согласия в форме записи (заявки сохраняются в БД — см. `global-spec.md` §14).
- **Security headers** настроены в `next.config.ts`: X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy.
- **A11y baseline** — семантические HTML-теги, alt-тексты, видимые focus-стили, контраст WCAG AA. shadcn/ui обеспечивает большую часть.
- **Lighthouse ≥90 (mobile)** — целевой таргет качества production-сборки.

## Workflow с Claude

- **Сначала обсуждение, потом действия.** Никаких изменений файлов без явного разрешения пользователя.
- **Работаем строго в папке проекта** — не выходим за пределы `/Users/saidakhmed/Desktop/Albident`.
- Перед работой над любой фичей или страницей — пройти процедуру skill `albident-spec` (`.claude/skills/albident-spec/SKILL.md`): прочитать релевантную спеку, сверить запрос и согласовать расхождения **до** написания кода. Если фича расходится со спекой — сначала обновить спеку, после — код.
- Любое изменение внешнего вида или новый UI-элемент — через skill `albident-design-map` (`.claude/skills/albident-design-map/SKILL.md`): сначала правим деталь в графической карте `design-map/index.html`, согласуем с пользователем, и только после явного «да» — переносим в реальный код сайта. Карта — единый источник правды по UI.
- Для бага — найти корневую причину, не маскировать `try/catch` или фолбэками.
- Не добавлять зависимости без нужды; перед `npm install <pkg>` — обосновать.
- Тексты UI — без канцелярита и медицинского жаргона.

## Plans workflow

Каждый план разработки оформляется как markdown-файл в `plans/`. Папка `plans/active/` — планы в работе, `plans/completed/` — завершённые (архив).

### Имя файла

`plans/active/<slug>.md`. Slug совпадает со slug ветки → пример: `booking-form.md` ↔ ветка `feature/booking-form`. Один источник правды для имени.

### Шаблон плана

```markdown
---
name: booking-form
branch: feature/booking-form
status: active
created: 2026-05-12
---

# План: Форма записи на приём

## Цель
Что должно стать возможным после этой работы (1–2 строки).

## Контекст
Ссылка на соответствующий feature-spec и важные оговорки.

## Шаги
- [ ] Шаг 1
- [ ] Шаг 2

## Критерии готовности
- По чему мы поймём, что план можно переносить в completed.
```

Структура рекомендуется, не энфорсится жёстко. Для мелких правок (опечатки, переименования, правки на одну строку) план в файле не создаётся вовсе. Для **мета-задач** (правки самого workflow, документации, инфраструктурных конфигов) plan-файл опционален — создаётся по желанию, если задача нетривиальна или хочется чек-лист.

### Жизненный цикл

1. Пользователь просит план → Claude создаёт `plans/active/<slug>.md`, согласует содержимое.
2. Создаётся ветка `<type>/<slug>` (см. GitHub workflow §1). Сам файл плана коммитится первым коммитом в ветке.
3. Разработка: коммиты в ветке, чек-боксы в `## Шаги` отмечаются по ходу.
4. Пользователь тестирует. Когда говорит «push» / «коммить и пушь» — это сигнал, что план завершён.
5. **Перед push Claude обязан** (одним отдельным коммитом, не смешивая с коммитами разработки):
   - `git mv plans/active/<slug>.md plans/completed/<slug>.md`
   - В frontmatter: `status: active` → `status: completed`, добавить `completed: <дата>`.
   - Если задача закрывает пункты из `roadmap/roadmap.md` — отметить их ✅, пересмотреть «Сейчас в работе» и «Ближайшие приоритеты», заменить `Обновлено: <дата>`. Если применимых пунктов нет (например, мета-задача) — обновить только дату.
   - Коммит: «План <slug> перенесён в completed».
6. Далее — стандартный GitHub workflow §3 (push + PR). После merge план остаётся в `plans/completed/` как часть истории репо.

Если оказалось, что план в работе устарел или отменяется, **не удалять** — переносить в `completed/` со `status: cancelled` и пометкой в теле, чтобы остался след решения.

## GitHub workflow

Жёсткое правило цикла «план → ветка → PR → merge → main». Соблюдается всегда, без исключений.

### 1. Старт нового плана → новая ветка

- Любая работа по конкретному плану/задаче ведётся **только в отдельной ветке**, не в `main`.
- К этому моменту файл плана `plans/active/<slug>.md` уже создан и согласован (см. Plans workflow).
- **До первой правки кода:**
  1. Убедиться, что текущая ветка — `main` и рабочее дерево чистое (`git status`).
  2. Подтянуть свежий `main`: `git fetch origin && git pull --ff-only origin main`.
  3. Создать ветку: `git checkout -b <type>/<slug>`. Slug — тот же, что и у файла плана.
- Формат имени ветки:
  - `feature/<slug>` — новая функциональность (напр. `feature/booking-form`).
  - `fix/<slug>` — исправление бага.
  - `chore/<slug>` — рутина (конфиги, зависимости, скаффолд).
  - `docs/<slug>` — только документация/спеки.
- Slug — короткий, латиница, kebab-case, без дат и номеров задач.

**Исключение: мелкие правки документации.** Можно сделать прямо в `main` без отдельной ветки, если правка **одновременно**:

- Касается только файлов документации (`*.md` в корне, `specifications/`, `plans/`, `roadmap/`, `.claude/skills/`).
- Это локальная мелкая операция: исправление опечатки, обновление даты / статуса / ссылки, удаление устаревшей строки, удаление одного устаревшего файла.
- Не вводит новых правил, разделов или концепций.

Любые другие изменения, в том числе любой код, — всегда через отдельную ветку.

### 2. Разработка

- Все коммиты — в созданной ветке. Промежуточные коммиты допустимы, но осмысленные (по логическим шагам, не «wip»).
- Сообщения коммитов — **на русском, без conventional-prefix** (см. «Соглашения по контенту»).

### 3. Завершение → push + PR (только по команде пользователя)

Когда пользователь говорит «закоммить и запушь» / «делай PR» / эквивалент:

1. **Перенести план в completed** — см. Plans workflow §5. Отдельный коммит до основного push. Если plan-файла не было (мета-задача или мелкая правка) — шаг пропускается.
2. `git status` + `git diff` — показать, что попадёт в коммит.
3. Сделать коммит (если ещё не сделан): `git add <конкретные файлы>` (не `-A`), затем `git commit`.
4. `git push -u origin <branch>`.
5. `gh pr create` с базой `main`. Заголовок — короткий (≤70 символов), тело — `## Summary` + `## Test plan`.
6. Вернуть пользователю URL PR.

Никаких `--force`, `--no-verify`, `--amend` на уже запушенные коммиты — без явной просьбы.

### 4. Merge — за пользователем

- **Claude НЕ мерджит PR.** Merge всегда делает пользователь вручную на GitHub.
- Перед merge пользователь сам ревьюит diff во вкладке **Files changed**. По умолчанию автоматическое многоагентное ревью не запускается — пользователь может попросить `/ultrareview` отдельно для крупных или рискованных PR.
- После создания PR — стоп. Ждём.

### 5. Продолжение работы → обязательный возврат на main

Когда пользователь говорит «продолжаем» / «следующий план» / просит начать новую задачу — **до обсуждения нового плана и до создания новой ветки**:

1. `git checkout main`.
2. `git pull --ff-only origin main` — подтянуть смердженный PR.
3. Опционально: `git branch -d <merged-branch>` — удалить локальную ветку, если её PR смерджен.
4. Только после этого — переходить к обсуждению нового плана и шагу 1 цикла.

Если рабочее дерево грязное или ветка не смерджена — остановиться и спросить пользователя, **не** делать `reset --hard` / `checkout -- .` / `branch -D` самовольно.
