# albident-spec

Цель: гарантировать, что любая работа над фичами Albident начинается с чтения соответствующей спецификации, а расхождения между запросом и спекой обнаруживаются и проговариваются **до** написания кода.

## Когда срабатывает

Триггер — запрос на работу над фичей или страницей. Примеры:

- «сделай форму записи», «давай главную», «запили чат»
- «добавь поле X в booking», «поменяй блок Hero»
- «как лучше устроить страницу услуги»
- «нужно подключить Resend для писем»

Не срабатывает на: правка опечатки, переименование переменной, вопрос о Next.js/TS в отрыве от фичи, обсуждение скоупа без действий, мета-работа (этот skill, CLAUDE.md, память).

## Процедура

Выполнить строго по порядку. Не предлагать план или код до прохождения всех 4 шагов.

### 1. Определить релевантные спеки

По карте выше. Если не уверен — лучше прочитать одну лишнюю, чем пропустить нужную. Всегда дополнительно учитывать `global-spec.md` (scope — что в MVP, что нет).

### 2. Прочитать спеку(и) целиком

Не выборочно, не по оглавлению. Полное прочтение через инструмент Read. Если уже читал ту же спеку в **этой же сессии** — можно опираться на прочитанное, не перечитывать.

### 3. Сверить запрос со спекой и зафиксировать результат

Сообщить пользователю **короткой выжимкой**:

- Какую спеку прочитал.
- Что в ней сказано про эту фичу (ключевые требования, ограничения, что в scope, что не в scope) — в 3–7 пунктах, не пересказ целиком.
- Один из трёх вердиктов:

**(а) Запрос совпадает со спекой.** → «Спека говорит X. Делаю по ней. Подтверди — начинаю.» Дальше — план реализации.

**(б) Запрос расходится со спекой.** → СТОП. Показать расхождение явно: «Ты просишь Y. Спека говорит Z. Расхождение в: …». Спросить: «Обновляем спеку (и тогда сначала правим её, потом код) или хочешь точечное отклонение в этой задаче с пометкой в коде?» Без ответа пользователя — код не писать.

**(в) Спека молчит про этот аспект.** → Проговорить явно: «Спека про это не говорит. Решение принимается вне спеки.» Предложить либо дополнить спеку (если вопрос архитектурный/долгосрочный), либо принять решение в рамках задачи (если вопрос локальный). Дождаться выбора пользователя.

### 4. После согласования — план реализации

Только теперь предлагать конкретные шаги, файлы, код. Если фича крупная — план шагами; если мелкая — сразу действие.

## Дополнительные проверки (всегда, при любом вердикте)

При сверке держать в голове ограничения проекта — они часто становятся причиной расхождений:

- **Не входит в MVP** (см. `global-spec.md`): онлайн-оплата, личный кабинет, блог, калькулятор стоимости, видео-консультации, CRM/МИС, карты, structured data, аналитика, captcha, полноценные KVKK/GDPR/ФЗ-152 тексты. Если запрос задевает это — почти всегда вердикт (б).
- **Фото реальных людей не используются** — врачи как силуэты с инициалами.
- **Соцсети** — только Instagram, WhatsApp, Telegram. Без Facebook.
- **Связь с пациентом** — через WhatsApp, не звонки и не email-цепочки.
- **Языки** — TR / EN / RU, по умолчанию RU, URL `/ru/...`.
- **Зависимости** — перед предложением `npm install <pkg>` обосновать через `technical-spec.md`.

## Формат сообщения после шага 3

Шаблон, по которому давать выжимку (адаптировать под фичу):

```
Читаю: <список спек>.

Спека говорит:
— <пункт 1>
— <пункт 2>
— <пункт 3>
…

Вердикт: <совпадает / расхождение / спека молчит>
<если расхождение или молчит — конкретный вопрос пользователю>
```

Коротко. Без копипасты больших кусков спеки. Пользователь сам читал спеку — нужна сверка, а не пересказ.

## Чего не делать

- Не цитировать спеку целиком — только релевантную выжимку.
- Не пропускать шаг 3 даже если кажется, что «и так понятно».
- Не писать код, пока пользователь не подтвердил план (при совпадении) или не разрешил расхождение/принял решение вне спеки.
- Не предлагать «компромиссные» решения, которые наполовину следуют спеке, наполовину нет — это нарушает правило «никаких слабых компромиссов». Либо по спеке, либо честное отклонение с обновлением спеки.
