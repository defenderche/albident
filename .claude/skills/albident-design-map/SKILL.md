# albident-design-map

Цель: графическая карта `design-map/index.html` — **единый источник правды по
UI** проекта Albident. Любое изменение внешнего вида и любой новый UI-элемент
сначала появляется в карте, проходит визуальное согласование с пользователем,
и только потом переносится в реальный код сайта (`components/`, `app/`).

Это даёт пользователю контроль над дизайном **до** того, как написан код
страниц и компонентов: он видит деталь в карте, одобряет или отклоняет.

## Когда срабатывает

Триггер — запрос, меняющий внешний вид или вводящий новый UI. Примеры:

- «сделай кнопку записи крупнее», «поменяй акцентный цвет на зелёный»
- «добавь на главную блок с отзывами в виде слайдера»
- «карточка услуги должна показывать иконку и рейтинг»
- «давай другой шрифт для заголовков», «больше скругления у карточек»
- «нужен новый компонент — баннер с акцией» (которого нет в карте)

Не срабатывает на:

- Правка текста/контента без изменения вида (заменить заголовок, описание,
  цену в `content/*.ts`).
- Бэкенд, Server Actions, валидация, i18n-ключи, логика чата.
- Баг без визуальной составляющей.
- Изменение **уже отражено** в `design-map/index.html` — тогда skill не нужен,
  сразу внедряем в код 1:1 с картой.
- Вопрос об устройстве Next.js/React в отрыве от конкретной UI-правки.
- Мета-работа: правки скиллов, `CLAUDE.md`, конфигов, инфраструктуры.

## Процедура

Выполнять строго по порядку. Не трогать реальный код сайта до шага 5.

### 1. Классифицировать запрос

- **(а) Изменение существующего элемента** дизайна (он есть в карте).
- **(б) Новый UI-элемент**, которого в карте нет.
- **(в) Не дизайн** → этот skill не нужен, работать как обычно.

Если запрос задевает не только вид, но и функционал / scope / структуру
контента — **сначала пройти `albident-spec`** (продуктовая спека — правда о том,
*что* делаем), затем вернуться сюда (карта — правда о том, *как выглядит*).
Порядок: спека → design-map → сайт.

### 2. Найти место в карте

Открыть `design-map/index.html`, найти релевантный токен / атом / молекулу /
организм. Для нового элемента — определить, в какой слой он попадает
(Foundations / Atoms / Molecules / Organisms) и рядом с чем встанет.

### 3. Внести изменение ТОЛЬКО в design-map

Правка или добавление вносится **исключительно** в `design-map/index.html`.
Файлы `components/` и `app/` на этом шаге не трогаем.

- Менять токены — через CSS-переменные в `:root` (единый источник правды).
- Новый элемент — оформить как живой компонент в нужной секции, с `⚠`-пометкой,
  если он адаптирован под ограничение Albident (рисованные аватары врачей вместо
  фото реальных людей, fallback на силуэт; соцсети только
  Instagram/WhatsApp/Telegram; связь — WhatsApp; цены — диапазон USD).

### 4. Показать и согласовать

Открыть карту в браузере (`open design-map/index.html`) и **явно** описать, что
изменилось. Спросить вердикт: **согласовано / отклонить / правки**.

- Без явного «да» — к коду сайта **не переходим**.
- При «правки» или «отклонить» — дорабатываем карту и снова показываем. Цикл
  повторяется, пока пользователь не одобрит. Сайт всё это время не трогаем.

### 5. Только после «да» — внедрить в реальный сайт

После явного согласия перенести **ту же** деталь в код (`components/`, `app/`,
токены Tailwind/CSS) — 1:1 с картой, без расхождений. Карта и реальный сайт
после этого совпадают.

## Чего не делать

- Не менять реальный сайт до согласования карты.
- Не править одновременно карту и сайт в одном заходе без одобрения между ними.
- Не пропускать показ карты пользователю и не «угадывать» согласие — нужно явное.
- Не допускать расхождения карты и сайта: если что-то поменяли в коде напрямую —
  это нарушение порядка, вернуть синхронность через карту.

## Связь с другими скиллами

- **`albident-spec`** — продуктовые спеки (`specifications/`). Если дизайн-правка
  задевает функционал, scope или структуру контента — сначала `albident-spec`
  (спека), затем этот skill (карта), затем сайт.
