# План: Страница услуги `/services/[slug]`

## Цель

Динамическая страница `/services/[slug]` по `feature-spec-service-page.md` для всех 8 услуг — 7 блоков сверху вниз (Hero / Описание / Этапы / Цены / Врачи / FAQ / Финальный CTA). SSG для всех 8 URL, 404 для неизвестных slug, CTA → `/booking?service=<slug>`, метаданные по шаблону спеки §4. Параллельно создаётся инфра для будущего `/about` (`content/doctors.ts` + `DoctorCard`).

## Контекст

Основная спека — `specifications/feature-specs/feature-spec-service-page.md`.
Дополнительно — `feature-spec-services.md` (уже использовалась), `global-spec.md` §7 (8 услуг), §9 (цены USD), §12 (силуэты вместо фото врачей).

Решения, согласованные до старта:

- **Блок «Связанные врачи» включается в этот план** — создаём минимальный `content/doctors.ts` (5 placeholder-врачей) и базовый `DoctorCard` сейчас, чтобы страница услуги получилась полной (7/7 блоков). `/about` потом реюзит эту инфру.
- **Шрифт/палитра/Hero-изображение не меняются** — тот же дефолт, что в `home-page` и `services-page`. Дизайн-язык — отдельный план `feature/design-language`.
- **Контент** — placeholder во всех новых полях (fullDescription, stages, subProcedures, faq, врачи). Владелец заменит позже (`global-spec §15`).
- **Силуэт врача** — простой SVG-плейсхолдер (круг с инициалами), как требует `global-spec §12`.
- **CTA → `/booking?service=<slug>`** — query параметр будет читаться, когда появится `/booking` (отдельный план).

## Критерии готовности

- `/[locale]/services/[slug]` для всех 8 услуг рендерит 7 блоков (Hero / Описание / Этапы / Цены / Врачи / FAQ / Финальный CTA).
- Неизвестный slug (например `/services/typo`) → 404 страница.
- Карточки врачей: силуэт-SVG с инициалами + ФИО + роль + 2-3 достижения.
- CTA-кнопки на странице услуги ведут на `/[locale]/booking?service=<slug>` (страница `/booking` пока 404 — query будет читаться позже).
- `<title>` и `<meta description>` собираются по шаблонам спеки §4 для всех 8 услуг × 3 локали.
- Главная (`/ru`) и `/ru/services` после изменений визуально без регрессий.
- `npm run lint`, `npx tsc --noEmit`, `npm test`, `npm run build` — зелёные. CI на PR — зелёный.

## Изменения после первоначальной реализации

После того как страница услуги была реализована по исходному плану (7 блоков, включая «Связанные врачи» и финальный CTA), при просмотре скриншотов всплыло три проблемы:

1. **Одинокая `DoctorCard`** — на услугах с одним связанным врачом (например, «Хирургия» → только Мехмет Йылмаз) карточка стояла в первой ячейке грида `lg:grid-cols-3` с пустотой справа на две трети ширины. Внутри карточки смешивалось выравнивание (аватар + ФИО + роль по центру, достижения по левому краю), что выглядело несбалансированно.
2. **Фрагментация чтения.** Четыре отдельные секции (Подробное описание / Этапы / Цены / FAQ) с разными фонами и бордерами рвали read-flow. Пациенту, который заходит изучить услугу, удобнее непрерывный блок с под-заголовками, чем серия визуально разделённых секций.
3. **Дублирование CTA.** Финальный CTA-блок в конце страницы дублировал sticky header (всегда виден) и Hero CTA. Лишняя кнопка, лишний скролл, без явной конверсионной пользы.

### Что изменилось

- **Структура: 7 блоков → 2** (Hero + Описание услуги).
- **В Hero** добавилось изображение услуги (placeholder; владелец заменит) и одна строка «Услугу ведёт: ФИО — роль» вместо отдельной секции «Связанные врачи».
- **Описание услуги** — единый блок: лид-параграф (полное описание), затем под-заголовки `h3` «Этапы лечения» / «Цены» / «Частые вопросы» в одном контейнере, без меняющихся фонов между подсекциями.
- **Финальный CTA-блок** убран и со страницы услуги, и со страницы `/services`. На обеих остаётся sticky header; на странице услуги дополнительно — CTA в Hero. Финальный CTA-блок теперь живёт только на главной (`/`).
- `DoctorCard.tsx` как компонент **остаётся в кодовой базе** — он понадобится для блока «Команда» на странице `/about` (см. `feature-spec-about.md`). На странице услуги он не используется.

### Спеки, которые были обновлены

- `feature-spec-service-page.md` (1.2 → 1.3): полностью переписан §1, обновлён §2.
- `feature-spec-services.md` (1.1 → 1.2): удалён §1.3 «CTA-блок», обновлены §1 и §2.
- `functional-map.md` (1.4 → 1.5): §2.1 переписан про правила CTA в Hero и финального CTA-блока; §3 (footer) — упоминание финального CTA уточнено («только на главной»).

«Почему» этих изменений: фидбэк по реальным скриншотам после реализации показал, что MVP-страница, формально соответствующая спеке, в живом виде читается фрагментарно. Решение унифицировать read-flow и убрать дублирование CTA родилось из этого фидбэка.

### Вторая итерация фидбэка: левое выравнивание и CTA в конце

После первой переделки осталось два момента:

4. **Разрыв «центр → линия → лево».** Hero был центрирован, между Hero и Описанием стояла горизонтальная линия (border-b у Hero), а Описание шло по левому краю. Глаз спотыкался на каждом переходе.
5. **CTA в Hero неудобен в read-flow.** Пациент дочитал страницу до FAQ — и чтобы записаться, ему нужно прокручивать обратно наверх. Sticky header помогает, но искать кнопку в шапке после длинного чтения — лишний шаг.

### Что изменилось во второй итерации

- **Единый магазин-стиль в Hero.** Hero переведён на левое выравнивание (h1, описание, doctor-line, картинка — все в одной колонке `max-w-3xl`, точно совпадает по ширине с Описанием). Убран `border-b` между Hero и Описанием — переход через воздух, не через линию.
- **Кнопка CTA перенесена из Hero в конец описания.** Теперь Hero заканчивается строкой с врачами; кнопка живёт после FAQ, в общем потоке текста. Размер увеличен (`size="lg"` + `h-12 px-7 text-base`), чтобы заметнее завершить read-flow.
- Иконка `ArrowRight` сохранена для визуального cue «жми и иди».

### Спеки, обновлённые во второй итерации

- `feature-spec-service-page.md` (1.3 → 1.4): из §1.1 убран пункт «Кнопка-CTA», в §1.2 добавлен §1.2.5 «Большая кнопка "Записаться"» в конце read-flow. §2 «Точки выхода» переписан под новое расположение CTA.
- `functional-map.md` (1.5 → 1.6): §2.1 разведены три понятия — CTA в Hero, кнопка в конце read-flow, финальный CTA-блок (с акцентным фоном). Страница услуги попадает в «кнопка в конце read-flow».

«Почему» этих изменений: магазин-стиль (как в референсе с Porsche 911) даёт единый left-aligned read из одной колонки без переходов «центр/лево». Кнопка в конце убирает необходимость скроллить наверх по дочитыванию.
