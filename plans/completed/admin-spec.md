# План: Спека админ-панели (управление услугами)

## Цель
Описать в спеках админ-панель для владельца: закрытая страница `/admin` с входом
по логину/паролю, где владелец сам управляет услугами (добавить / изменить /
удалить). Услуги переезжают из `content/services.ts` в БД Supabase; сайт и
чат-ассистент читают их оттуда. Это фундамент — код пишется отдельными планами
после согласования спеки.

## Контекст
- Расхождение со спекой (вердикт «б» по skill `albident-spec`):
  - `technical-spec` §4 — контент в файлах, в БД только `bookings`.
  - `technical-spec` §1/§9 — Supabase хранит только `bookings`, anon-ключ не используется.
  - `technical-spec` §8 — услуги статические (SSG, `generateStaticParams`).
  - `global-spec` §13/§16 — исключён кабинет пациента; «менеджер» через dashboard.
- Решение: обновляем спеки (смена архитектуры контента, не точечное отклонение).
- Проект — учебный/портфолио (`global-spec` §1); цель админки — продемонстрировать
  навык (full-stack: авторизация + CRUD + БД), а не реальная эксплуатация.

## Решения (согласованы в обсуждении)
- Охват v1 — **только услуги**. Врачи/отзывы/FAQ — позже.
- Хранение услуг — **JSONB** (верхнеуровневые поля колонками, вложенное/мультиязычное в JSONB).
- Авторизация — **Supabase Auth** (email+пароль), один админ-пользователь.
- Переводы — **авто через OpenAI** (владелец пишет RU → EN/TR переводятся при сохранении, редактируемо).
- Slug — **авто из английского названия, заблокирован** после создания.
- Главная — услуга показывается там по флагу **«на главной»**.
- Дыра учтена: форма записи `/booking` и её валидация тоже читают услуги из БД (раньше зашитый перечень).
- Этапность реализации (отдельные планы/PR): (1) перенос услуг в БД и чтение из неё,
  (2) авторизация + каркас `/admin`, (3) CRUD-формы + авто-перевод + ревалидация.
